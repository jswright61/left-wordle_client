⏺ Load Impact for This Specific App

  First, an important framing point: this app's API is extremely light. Here's
  why:

  What actually hits the Python backend per user session

  A typical Wordle session generates roughly:
  - 1-2 calls on page load (pull game state, if logged in)
  - 1 call per guess to validate word (~6 max per game)
  - 1-2 calls on game completion (push state, update history)

  Total: ~10 API calls per 5-15 minute session

  That's roughly 0.7-2 requests/minute per active user.

⏺ Yes, absolutely - and passkeys are actually a better fit for this app than
  magic links. Here's why, and how to architect it.

  Why Passkeys Work Better Here

  The current magic link flow exists because there are no passwords - it's
  already "passwordless." Passkeys take that further:

  Column 1: User action
  Magic Link: Check email, click link
  Passkey: Touch ID / Face ID / Windows Hello
  ────────────────────────────────────────
  Column 1: Speed
  Magic Link: 30-60 seconds
  Passkey: ~1 second
  ────────────────────────────────────────
  Column 1: Works offline
  Magic Link: No
  Passkey: Credential stored locally
  ────────────────────────────────────────
  Column 1: Phishing resistant
  Magic Link: Partially
  Passkey: Yes (origin-bound by spec)
  ────────────────────────────────────────
  Column 1: Multi-device
  Magic Link: Via email each time
  Passkey: Syncs via iCloud/Google/1Password
  ────────────────────────────────────────
  Column 1: Server secret needed
  Magic Link: Yes (Postmark token)
  Passkey: No - eliminates Postmark entirely

  The sync across devices works because modern passkeys are synced credentials -
   iOS syncs via iCloud Keychain, Android via Google Password Manager, and
  cross-platform password managers (1Password, Bitwarden) also support them.

  ---
  The Architecture

  Browser                              FastAPI + Postgres
      │                                       │
      │  1. POST /auth/register/begin         │
      │  ─────────────────────────────────>   │  generate challenge, store in DB
      │  <─────────────────────────────────   │  return
  PublicKeyCredentialOptions
      │                                       │
      │  2. navigator.credentials.create()   │
      │     (local only - Touch ID fires)    │
      │                                       │
      │  3. POST /auth/register/complete      │
      │  ─────────────────────────────────>   │  verify attestation
      │  <─────────────────────────────────   │  store public key, return JWT
      │                                       │
      │  ─ ─ ─ ─ later logins ─ ─ ─ ─ ─     │
      │                                       │
      │  4. POST /auth/login/begin            │
      │  ─────────────────────────────────>   │  generate challenge
      │  <─────────────────────────────────   │  return challenge + allowed
  creds
      │                                       │
      │  5. navigator.credentials.get()      │
      │     (local only - biometric fires)   │
      │                                       │
      │  6. POST /auth/login/complete         │
      │  ─────────────────────────────────>   │  verify signature w/ stored
  pubkey
      │  <─────────────────────────────────   │  return JWT

  The critical security property: the private key never leaves the device. The
  server only ever sees a signed challenge - there's nothing to steal from the
  server that allows impersonation.

  ---
  Database Schema

  -- Users table (minimal - no password hash needed)
  CREATE TABLE users (
      id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      email       TEXT UNIQUE,          -- optional, for display only
      display_name TEXT,
      created_at  TIMESTAMPTZ DEFAULT NOW()
  );

  -- One user can have multiple passkeys (phone + laptop + YubiKey)
  CREATE TABLE passkey_credentials (
      credential_id   TEXT PRIMARY KEY,  -- base64url, from authenticator
      user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
      public_key      BYTEA NOT NULL,    -- COSE-encoded public key
      sign_count      BIGINT NOT NULL DEFAULT 0,  -- replay attack prevention
      device_type     TEXT,              -- 'platform' | 'cross-platform'
      backed_up       BOOLEAN DEFAULT FALSE,      -- synced to cloud keychain?
      transports      TEXT[],            -- ['internal','hybrid','usb','nfc']
      friendly_name   TEXT,              -- "Scott's iPhone" - user-editable
      created_at      TIMESTAMPTZ DEFAULT NOW(),
      last_used_at    TIMESTAMPTZ
  );

  -- Short-lived challenges (prevent replay attacks)
  CREATE TABLE webauthn_challenges (
      challenge   TEXT PRIMARY KEY,
      user_id     UUID REFERENCES users(id),  -- NULL during registration
      expires_at  TIMESTAMPTZ NOT NULL,       -- NOW() + 5 minutes
      purpose     TEXT NOT NULL               -- 'registration' |
  'authentication'
  );

  ---
  Python Stack

  fastapi          - web framework
  py_webauthn      - WebAuthn ceremony handling (Duo Security library)
  sqlalchemy       - ORM
  asyncpg          - async Postgres driver
  python-jose      - JWT session tokens
  pydantic         - request/response validation

  py_webauthn is the authoritative Python library - it's maintained by the FIDO
  Alliance community and handles all the cryptographic verification.

  ---
  Key API Endpoints

  # Registration (new user, or adding a new device)
  POST /api/auth/register/begin      # → returns
  PublicKeyCredentialCreationOptions
  POST /api/auth/register/complete   # → verifies, stores pubkey, returns JWT

  # Authentication
  POST /api/auth/login/begin         # → returns
  PublicKeyCredentialRequestOptions
  POST /api/auth/login/complete      # → verifies signature, returns JWT

  # Credential management (authenticated users only)
  GET  /api/auth/credentials         # → list user's passkeys + device names
  PATCH /api/auth/credentials/{id}   # → rename "Scott's iPhone 15"
  DELETE /api/auth/credentials/{id}  # → revoke a credential

  # Session
  POST /api/auth/logout
  GET  /api/auth/me

  ---
  Frontend (what changes in the browser)

  The browser side is surprisingly simple - the Web Authentication API is built
  into every modern browser:

  // Registration - ~15 lines replacing the magic link send flow
  async function registerPasskey(username) {
      // 1. Get options from server
      const { options } = await fetch('/api/auth/register/begin', {
          method: 'POST',
          body: JSON.stringify({ username })
      }).then(r => r.json());

      // 2. Browser prompts Touch ID / Face ID (no network call)
      const credential = await navigator.credentials.create({ publicKey: options
   });

      // 3. Send result to server
      const { token } = await fetch('/api/auth/register/complete', {
          method: 'POST',
          body: JSON.stringify(credential)
      }).then(r => r.json());

      // Done - user is now authenticated
  }

  You'd also want @simplewebauthn/browser (small npm package) to handle the
  base64url encoding that the raw API requires - saves about 50 lines of
  boilerplate.

  ---
  Fallback Strategy

  For the small percentage of users on older browsers or who lose access to
  their device:

  1. Primary: Passkey (Touch ID, Face ID, Windows Hello, synced credential)
  2. Fallback: Email magic link (keep Postmark, just not the primary flow)
  3. Recovery: "Add a new passkey" flow triggered via email link

  This means you still need Postmark in .env, but it's only used for account
  recovery - not for the normal login path.

❯ Does Windows provide passkey syncing between Windows machines? What About
   Linux? With either of the two OSes can passkeys be synced to phones or
  would 1Password or BitWarden be needed for sync? For those who cannot
  sync, the email would be used to register multiple passkeys for a given

⏺ Windows

  Syncing between Windows machines: Yes, but recently and with caveats. Windows
  11 24H2 (October 2024) introduced a native passkey manager that syncs via
  Microsoft Account across Windows devices. Before that, Windows Hello passkeys
  were device-bound to the TPM chip - no sync at all.

  Windows to Android/iPhone: Not natively. However, there's a useful middle
  ground called hybrid transport (CTAP2): your phone acts as an authenticator
  for your Windows machine via Bluetooth proximity + QR code. You scan a QR code
   on Windows with your phone, your phone's biometric fires, and the login
  completes on Windows. But this is a one-time authentication, not a synced
  credential - the passkey lives on the phone and Windows just proxies to it
  each time.

  So practically: if someone has their passkey in Google Password Manager on
  Android, they can use it on Windows Chrome, but it requires the phone to be
  nearby each time.

  ---
  Linux

  No native sync whatsoever. GNOME Keyring and KWallet have no passkey cloud
  sync. Passkeys created on Linux are device-bound, full stop.

  The realistic options for Linux users are:
  - 1Password or Bitwarden desktop apps (cross-platform sync)
  - Hardware key (YubiKey, etc.) - physical "sync" via carrying the key
  - Android/iPhone via hybrid transport - same QR code approach as Windows

  Linux is the weakest point in the passkey ecosystem right now.

  ---
  Your Multi-Passkey Question

  Yes, exactly right - and this is the standard recommended pattern in the
  WebAuthn spec. The flow would be:

  First device (e.g. iPhone):
    → Register passkey → stored in iCloud Keychain

  New device (e.g. Linux laptop):
    → Log in via email magic link (one-time fallback)
    → Prompt: "Register a passkey for this device?"
    → Register second passkey → stored locally on Linux

  Third device (Windows):
    → Log in via email magic link (or use iPhone via hybrid transport)
    → Register third passkey → stored in Microsoft Account

  Each passkey is independently registered to the same user_id in your
  passkey_credentials table - that's exactly what the multi-credential schema is
   designed for. The credential management UI (GET /api/auth/credentials) then
  lets users see and name all their registered devices:

  Your passkeys:
    📱 Scott's iPhone 15       Added Jan 2025   Last used today    [Remove]
    🐧 Linux workstation       Added Jan 2025   Last used 3 days ago [Remove]
    🪟 Windows laptop          Added Feb 2025   Last used 1 week ago [Remove]

  This is precisely how GitHub, Apple, and Google implement it. The email magic
  link never disappears - it just becomes the bootstrap mechanism for new
  devices rather than the primary login flow.

❯ And just to be clear, that passkey first approach would be harder to
  implement in the current system?

⏺ Yes, significantly harder - and for a fundamental reason, not just a
  complexity reason.

  WebAuthn requires server-side state. The challenge-response ceremony cannot be
   done client-only:

  Current architecture:          What passkeys need:

  Browser ←→ Supabase           Browser ←→ Your Server ←→ Database
    (no middle layer)               (challenge storage,
                                     signature verification,
                                     public key storage)

  The server has to:
  1. Generate a random challenge and store it temporarily
  2. Receive the signed response from the authenticator
  3. Cryptographically verify the signature against the stored public key
  4. Track the sign counter to prevent replay attacks

  None of that can happen in the browser.

  Your options in the current system would be:

  1. Use Supabase's passkey support - Supabase did add experimental passkey
  support, but you'd still have the anon key exposed in client code, still be
  locked into Supabase, and still have the other problems you wanted to solve.
  You'd be trading one dependency for another.
  2. Deploy a separate backend just for auth - messy split architecture where
  auth lives on your server but sync still goes through Supabase.

  Either way, you end up needing a backend - which means the current "no
  backend" architecture breaks down the moment you add passkeys.

  The FastAPI migration and passkey-first auth are actually the same project.
  The backend you need for passkeys is the same backend that replaces Supabase
  for sync, hides the word lists from the browser, and keeps secrets out of the
  repository. You build it once.

❯ A major reason I currently have a local storage first approach is the
  hassle and privacy implications of requiring a login, but were I to
  implement passkeys a user who doesn't want to use their real email
  address can have full login functionality with just passkeys, what they
  would lose is email fallback for recovery and adding devices, but if they
   could sync the passkeys on their own they could still have multiple
  devices. Am I thinking about this correctly? And a cloud first data store
   is much simpler and less prone to sync issues and would still allow for
  offline play and later sync up (even though offline play is not an
  important design criteria of mine)

⏺ Yes, your thinking is exactly right on all points. Let me confirm each one:

  Anonymous Passkey Registration

  Correct. WebAuthn has no requirement for an email address - the protocol only
  needs a userHandle (a UUID you generate) and a displayName (can be anything,
  even "Anonymous Player"). A privacy-conscious user could register as:

  {
    "userHandle": "a3f8c2d1-...",
    "displayName": "Player"
  }

  No email ever collected. Their data lives server-side under that UUID, their
  passkey syncs via iCloud/Google/Bitwarden, and they have full multi-device
  functionality. The email field in your users table would simply be nullable.

  The tradeoff is explicit and the user owns it: no email = no recovery path.
  That's a reasonable choice to offer people.

  ---
  Cloud-First is Genuinely Simpler

  You're right that the local-first approach is the source of most of your
  current complexity. Look at what it costs you today:

  - supabase-sync.js is 1,013 lines - almost entirely conflict resolution logic
  - Pre-merge backups, device ID tracking, debounced pushes, dirty row detection
  - Edge cases for "what if the same puzzle completed differently on two
  devices"

  Cloud-first with a server as the single source of truth eliminates all of
  that. The architecture becomes:

  On page load:     fetch state from server → render
  On guess:         POST to server → update UI with response
  On network loss:  queue writes locally → replay on reconnect
  Offline read:     serve cached state from localStorage

  That last point - the offline queue - is maybe 50 lines of straightforward
  code. No merge logic needed because the server is always authoritative;
  offline writes just wait their turn.

  ---
  The Design That Falls Out of This

  Users table
    - id (UUID)          ← always present
    - email (nullable)   ← only if they want recovery
    - display_name

  Passkey credentials    ← one or more per user
    - tied to user.id

  Game data              ← server-authoritative
    - tied to user.id
    - localStorage is just a read cache

  Three tiers of user by their own choice:

  User type: Anonymous
  Email?: No
  Recovery?: No
  Multi-device?: Only if they sync passkeys themselves
  ────────────────────────────────────────
  User type: Privacy-first
  Email?: No
  Recovery?: No
  Multi-device?: Via their own passkey sync
  ────────────────────────────────────────
  User type: Full account
  Email?: Yes
  Recovery?: Yes (magic link)
  Multi-device?: Via email bootstrap + passkey sync

  No one is forced into anything, and the complexity of your system doesn't
  increase for any of the three cases - the difference is just a nullable email
  field.

  ---
  The other thing worth noting: since offline play isn't a design priority, you
  could go fully server-authoritative from the start and not even build the
  offline queue initially. Add it later if users ask for it. That's probably 80%
   less code than the current sync system.
